# Oh My OpenAgent 实战搭配使用指南

> 上一篇文章讲了配置文件里每一项是什么。这篇讲**怎么用**——日常工作中如何搭配这些 AI 特工，什么场景走什么流程。

---

## 一、先理解核心工作流

OmO 有三种工作模式，由简到繁：

```
你有一个任务
    │
    ├─ 简单小活？ → 直接说话，不用任何特殊操作
    │
    ├─ 复杂但不想废话？ → 输入 "ulw" 或 "ultrawork"
    │
    └─ 复杂且需要精确控制？ → @plan 规划 → /start-work 执行
```

这张图就是你日常使用的全部决策逻辑。下面逐一展开。

---

## 二、模式一：直接对话（日常 80%）

### 什么时候用

- 改一个变量名
- 修一个类型错误
- 问「这个函数是干什么的」
- 写一个小工具函数
- 加一行日志

### 怎么用

直接在终端里说话，和普通 AI 对话一样：

```
我：帮我把 src/utils/format.ts 里的 formatDate 函数名改成 formatISODate
我：这个 API 返回 500，帮我看看哪里错了
我：给我解释一下 auth middleware 的逻辑
```

### 背后发生了什么

OmO 的 **IntentGate（意图门）** 会先分析你真正的意图——是研究？是修改？是修复？然后自动路由到合适的 agent 和模型。你不需要手动指定。

> **你目前的配置**：所有 agent 都用 `gpt-5-nano`，所以路由到谁差别不大。但这套流程本身是完整的。

---

## 三、模式二：ultrawork（懒人神器）

### 什么时候用

- 任务复杂，但**不想写长篇需求**
- 「你帮我搞定」类型的任务
- 信任 AI 自主探索和决策
- 任务边界清晰但步骤多

### 怎么用

就一个词：

```
ulw
```

或者完整版：

```
ultrawork
```

后面可以跟具体任务描述：

```
ulw 给 API 接口加上 JWT 认证
ulw 修复所有失败的测试
ulw 把配置文件从 JSON 迁移到 YAML
```

### 背后发生了什么

```
你输入 "ulw"
    │
    ▼
IntentGate 分析意图
    │
    ▼
Sisyphus（总指挥）接手
    │
    ├─ 派 Explore 搜索代码库，找到相关文件
    ├─ 派 Librarian 查 JWT 最佳实践文档
    ├─ 派 Hephaestus 执行具体修改
    ├─ 用 LSP 检查有没有报错
    └─ 循环直到完成，不半途而废
```

### 为什么它不会停

OmO 有两个机制保证任务一定做完：

1. **Todo Enforcer（任务强制执行器）**：agent 一旦有空闲的 todo，系统会自动提醒它继续干活
2. **Ralph Loop**：自我驱动的循环，直到 100% 完成

### 实战例子

```
# 例子 1：修复测试
ulw 修复 user.service.test.ts 里的 3 个失败用例

# OmO 会：
# 1. 跑测试看报什么错
# 2. 找到对应的源码
# 3. 分析是测试写错了还是源码有 bug
# 4. 修改、再跑测试
# 5. 直到全部通过
```

## 模式三：@plan → /start-work（精确控制流）

这是 OmO 最强大的工作模式，适合多步骤、跨多天的大任务。

### 什么时候用

- 大型重构（改几十个文件）
- 新功能开发（跨越多个模块）
- 架构变更（影响面广）
- 需要留下决策记录的任务
- 跨天/跨会话的任务（今天做不完明天继续）

### 完整流程

#### 第一步：启动规划

有两种方式进入规划模式：

**方式 A：按 Tab 切换 agent**

```
1. 在终端按 Tab 键
2. 从列表中选择 "Prometheus"（普罗米修斯，战略规划师）
3. 描述你的任务
```

**方式 B：用 @plan 命令（更快捷）**

```
@plan 重构用户认证系统，把 session 改成 JWT
```

#### 第二步：接受采访

Prometheus 不会直接动手。它会像真正的工程师一样采访你：

```
Prometheus：你希望 JWT 的过期时间设多长？
你：access token 15 分钟，refresh token 7 天

Prometheus：现有的 session 表要保留吗？还是直接删掉？
你：保留，做平滑迁移

Prometheus：测试策略？单元测试 + 集成测试？
你：都要

Prometheus：有没有绝对不能改的模块？
你：不要动 /api/v1/legacy 下面的任何东西
```

这个过程确保你们对需求的理解一致，避免 AI 瞎猜。

#### 第三步：计划审查

Prometheus 采访完后，会自动走审查流程：

```
Prometheus 起草计划
    │
    ▼
Metis（漏洞分析师）检查：
  - 有没有遗漏的需求？
  - 有没有模糊的地方？
  - 有没有过度设计？
    │
    ▼
计划写进 .omo/plans/jwt-migration.md
    │
    ▼
（可选）Momus（严格审查员）按 4 个标准打分：
  - 清晰度：每个任务有没有指明具体文件？
  - 可验证性：验收标准能不能量化？
  - 上下文：执行者拿到的信息够不够？
  - 大局观：目标、背景、流程清不清楚？
```

#### 第四步：开始执行

计划确认后：

```
/start-work
```

这会触发 Atlas（执行指挥）启动：

```
Atlas 读取计划
    │
    ├─ 任务 1：创建 JWT 工具函数 → 派给 Sisyphus-Junior（category=quick）
    │   └─ 完成，提取经验：项目用 bcrypt 做哈希，密钥从环境变量读
    │
    ├─ 任务 2：改造 auth middleware → 派给 Sisyphus-Junior（category=deep）
    │   └─ 完成，提取经验：middleware 遵循 Express 标准签名
    │
    ├─ 任务 3：写测试 → 派给 Sisyphus-Junior（category=quick）
    │   └─ 完成，提取经验：测试框架是 vitest
    │
    └─ 全部完成，Atlas 生成最终报告
```

> **关键机制：智慧积累（Wisdom Accumulation）**
> 
> 任务 1 中发现的经验（比如「项目用 bcrypt」）会自动传给任务 2、3。后面的特工不会重复犯错，也不用重新探索代码库。这就是 OmO 比单个 AI 强的地方——**团队协作 + 经验传递**。

#### 第五步：跨天继续

如果你今天没做完，关电脑了。明天打开 OpenCode：

```
/start-work
```

Atlas 会读取 `.omo/boulder.json`（任务进度文件）：

```
"恢复任务 'JWT 认证迁移' - 已完成 3/8 个任务"
从任务 4 继续...
```

**完全不需要重新解释上下文。** 这是 OmO 最实用的功能之一。

## 四、什么时候用 Hephaestus

Hephaestus 是 GPT 原生的深度工作者。他不像 Sisyphus 那样依赖详细计划，而是自己探索、自己决策。

### Hephaestus vs ulw 怎么选

| 场景 | 用什么 | 原因 |
|------|--------|------|
| 加个 API 校验、改个配置 | `ulw` | Sisyphus 够用 |
| 设计新插件系统 | 切到 Hephaestus | GPT-5.5 架构推理更强 |
| 调试复杂的竞态条件 | 切到 Hephaestus | 需要长链推理 |
| 迁移数据库（Mongo→PG） | 切到 Hephaestus | 跨领域知识综合 |
| 修失败的测试 | `ulw` | 步骤明确，Sisyphus 更快 |
| 重构单体到微服务 | 切到 Hephaestus | 需要 GPT 的推理风格 |

### 怎么切换到 Hephaestus

按 `Tab`，选 `Hephaestus`，然后直接描述你的目标（不是步骤）：

```
❌ 错误：帮我修改 src/auth.ts 的第 15 行，把 session 改成 jwt...
✅ 正确：把认证系统从 session 迁移到 JWT，保持向后兼容
```

Hephaestus 不需要你一步步教，给他目标就行。

## 五、什么时候用 Oracle

Oracle 是**只读**顾问。他不会改代码，只提供建议。

### 典型场景

```
我：Oracle，我打算用 Redis 做缓存层，但现在系统用的是内存缓存。
    有什么坑需要注意？

Oracle：（分析代码库后）
  1. 你们的缓存 key 命名不规范，迁移前先统一
  2. UserService 里有 3 处直接操作缓存的代码，需要抽一个 CacheService
  3. 建议用 ioredis 而不是 node-redis，因为你们用了 cluster 模式
  4. 内存和 Redis 双写的过渡期至少保留 2 周
```

### 怎么用

按 `Tab` 选 `Oracle`，或者在 Sisyphus 中：

```
@oracle 帮我评估一下这个 PR 的架构风险
```

## 七、进阶：Team Mode（团队模式）

Team Mode 默认关闭。开启后，一个 lead agent 可以同时指挥最多 8 个特工并行工作。

### 开启方式

在 `oh-my-openagent.json` 中添加：

```jsonc
{
  "team_mode": {
    "enabled": true,
    "max_parallel_members": 4,
    "tmux_visualization": true  // 可以在 tmux 里看到每个特工的工作画面
  }
}
```

### 什么时候开

- 大规模并行探索（同时搜 4 个模块）
- 安全审计（3 个漏洞猎人 + 2 个 PoC 工程师并行）
- 大型重构（多个模块同时改）

### 什么时候不开（默认不开就对了）

- 日常小任务：杀鸡不用牛刀
- 单线程任务：没必要开
- 成本敏感：并行 = 多倍 token 消耗

## 八、常见问题

### Q: ulw 和 @plan 到底怎么选？

| ulw | @plan |
|-----|-------|
| 你懒得写需求 | 你需要精确控制 |
| 任务边界清晰 | 任务影响面广 |
| 信任 AI 自主决策 | 需要留下决策记录 |
| 一次性能做完 | 可能跨多天 |
| 「帮我搞定这个」 | 「先规划，再执行」 |

### Q: 我用 ulw 但它做错了怎么办？

在 Sisyphus 里直接说「不对，XXX 应该是 YYY」，它会修正。ulw 不是一次性黑箱，你可以中途介入。

### Q: @plan 出来的计划我不满意？

直接告诉 Prometheus「这个计划不好，因为 XXX」。它会重新采访你，调整计划。你也可以让 Momus 严格审查。

### Q: 我全部用 gpt-5-nano，有必要搞这么复杂吗？

有。即使模型相同，**流程本身**也很有价值：
- `ulw` 的自主循环机制让你省心
- `@plan` 帮你理清思路
- `/start-work` 的进度恢复让你跨天不丢上下文

模型只是引擎，流程才是驾驶方式。

### Q: 我想换个更好的模型怎么办？

编辑 `oh-my-openagent.json`，比如给 oracle 换一个更强的模型：

```jsonc
"oracle": {
  "model": "openai/gpt-5.5",
  "variant": "high"
}
```

其他 agent 保持不变。改完重启 OpenCode 生效。
