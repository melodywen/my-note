# 08 分布式训练：DDP 与 DeepSpeed 深度对比

> 本篇深入讲解 LLaMA Factory / ms-SWIFT 等框架里「分布式类型」选项中 **DDP** 和 **DeepSpeed** 的本质区别、工作原理、以及怎么选。一句话结论：**小模型(4B/8B)多卡用 DDP 就够了；大模型(>30B)或显存不够时用 DeepSpeed。**

## 1. 为什么需要分布式训练？

先搞清楚「问题是什么」，才能理解两个方案的差异。

### 1.1 核心矛盾：模型太大，一张卡放不下

训练大模型时面临两个瓶颈：

| 瓶颈       | 表现                      | 解法                          |
| -------- | ----------------------- | --------------------------- |
| **显存不够** | 模型权重 + 梯度 + 优化器状态超过单卡显存 | 模型并行 / 显存优化（DeepSpeed ZeRO） |
| **训练太慢** | 数据量大，单卡跑几天几夜            | 数据并行（DDP / DeepSpeed 都支持）   |

### 1.2 两种并行策略

| 策略 | 思路 | 适合场景 |
|---|---|---|
| **数据并行(Data Parallel)** | 模型复制多份，数据切分，各自算梯度再同步 | 模型能放进单卡，想加速训练 |
| **模型并行(Model Parallel)** | 模型切分放到不同卡上，一张卡只放一部分 | 模型太大单卡放不下 |

> [!info] 本篇重点讲**数据并行**的两种实现（DDP vs DeepSpeed），因为 LLaMA Factory 的分布式选项里这两个都是数据并行方案。模型并行（Pipeline Parallel / Tensor Parallel）是另一回事，DeepSpeed 也支持但不在这俩选项里。

## 2. DDP (DistributedDataParallel)

### 2.1 一句话定义

**DDP 是 PyTorch 原生的数据并行方案**：每张 GPU 持有完整的模型副本，数据切分后各自前向+反向，最后通过 `all-reduce` 同步梯度。

### 2.2 工作原理（一步一步）

```
初始状态：每张卡都有完全相同的模型副本

┌─────────┐  ┌─────────┐  ┌─────────┐
│  GPU 0  │  │  GPU 1  │  │  GPU 2  │
│ 完整模型  │  │ 完整模型  │  │ 完整模型  │
└─────────┘  └─────────┘  └─────────┘

训练一步（forward + backward）:
  GPU 0: 拿到 batch[0:32],   前向→损失→反向→得到梯度 g0
  GPU 1: 拿到 batch[32:64],  前向→损失→反向→得到梯度 g1
  GPU 2: 拿到 batch[64:96],  前向→损失→反向→得到梯度 g2

all-reduce 同步（关键步骤）:
  GPU 0: g = (g0+g1+g2)/3, 更新自己的模型
  GPU 1: g = (g0+g1+g2)/3, 更新自己的模型
  GPU 2: g = (g0+g1+g2)/3, 更新自己的模型

→ 三步后，三张卡的模型权重又完全一致了
```

### 2.3 显存占用分析（每张卡）

以 **7B 模型、AdamW 优化器、FP16 混合精度** 为例：

| 内容 | 占用 | 说明 |
|---|---|---|
| 模型参数(FP16) | ~14 GB | 7B × 2 bytes |
| 梯度(FP16) | ~14 GB | 7B × 2 bytes |
| 优化器状态(FP32) | ~28 GB | 7B × 4 bytes × 1（仅 master weights，Adam 实际是 2 份≈56GB） |
| 激活值 | 取决于 batch size | 临时开销 |
| **合计(全量微调)** | **~70~100 GB** | 单卡 80GB(A100) 才够 |

> [!warning] LoRA 微调时显存压力小很多
> LoRA 只训练少量参数（约全参数的 0.1%~1%），优化器状态也只存 LoRA 部分的。7B 模型 LoRA 微调，单卡 24GB(3090/4090) 通常够用。

### 2.4 优点

| 优点 | 说明 |
|---|---|
| **简单** | PyTorch 原生支持，`torchrun` 或 `accelerate` 一行命令就能起 |
| **通信开销小** | 只需 `all-reduce` 同步梯度，通信量是 `2×模型参数`（一次 reduce，一次 broadcast） |
| **训练速度快** | 通信量少 → 每个 step 耗时短 |
| **稳定** | PyTorch 内置，bug 少，社区支持好 |

### 2.5 缺点

| 缺点 | 说明 |
|---|---|
| **显存效率低** | 每张卡存完整模型副本，**冗余存储**严重 |
| **模型大小受限于单卡显存** | 模型必须能放进一张卡，否则 DDP 直接 OOM |
| **卡数越多冗余越大** | 8 张卡存 8 份完整模型，浪费严重 |

### 2.6 适用场景

- ✅ **模型能放进单卡**（如 4B/8B LoRA 微调，甚至 7B 全量微调在 80GB 卡上）
- ✅ **多卡只是为了加速训练**（数据并行）
- ✅ **追求简单、稳定、快速**

## 3. DeepSpeed

### 3.1 一句话定义

**DeepSpeed 是微软开发的深度学习优化库**，核心功能是 **ZeRO（Zero Redundancy Optimizer）**——通过把模型参数、梯度、优化器状态**分片（shard）存储**到不同 GPU 上，极度压缩显存占用，让小卡也能训大模型。

### 3.2 ZeRO 的三个阶段（核心！）

ZeRO 的本质：**不要每张卡存全部，改为每张卡只存一部分，需要时再临时拼回来。**

#### ZeRO Stage 1：分片优化器状态

```
Before Stage 1 (DDP 模式):
  GPU 0: 优化器状态[全部]  ← 浪费！
  GPU 1: 优化器状态[全部]  ← 浪费！
  GPU 2: 优化器状态[全部]  ← 浪费！

After Stage 1:
  GPU 0: 优化器状态[1/3]
  GPU 1: 优化器状态[1/3]
  GPU 2: 优化器状态[1/3]
  → 每张卡优化器状态减少为 1/N
```

- **省什么**：只分片优化器状态（Adam 的 momentum + variance）
- **省多少**：约 4× 节省（Adam 优化器状态是模型参数的 2 倍 FP32）
- **模型参数/梯度**：仍然每张卡完整存储

#### ZeRO Stage 2：分片优化器状态 + 梯度

```
After Stage 2:
  GPU 0: 优化器状态[1/3] + 梯度[1/3]
  GPU 1: 优化器状态[1/3] + 梯度[1/3]
  GPU 2: 优化器状态[1/3] + 梯度[1/3]
```

- **省什么**：优化器状态 + 梯度都分片
- **省多少**：约 8× 节省
- **模型参数**：仍然每张卡完整存储
- **通信**：反向时做 `reduce-scatter` 替代 `all-reduce`，通信量减半

#### ZeRO Stage 3：分片优化器状态 + 梯度 + 模型参数（完全分片）

```
After Stage 3:
  GPU 0: 优化器状态[1/3] + 梯度[1/3] + 模型参数[1/3]
  GPU 1: 优化器状态[1/3] + 梯度[1/3] + 模型参数[1/3]
  GPU 2: 优化器状态[1/3] + 梯度[1/3] + 模型参数[1/3]
```

- **省什么**：全部三样都分片（最彻底）
- **省多少**：约 N× 节省（N = GPU 数量）
- **关键机制**：前向/反向时通过 `all-gather` **临时重建**当前层所需的参数，用完即释放
- **代价**：通信量大增（all-gather + reduce-scatter 各 N 次）

#### ZeRO Stage 3 + Offload：把数据卸载到 CPU/NVMe

```
Stage 3 + CPU Offload:
  GPU: 模型参数[当前层]  ← 只放正在算的层
  CPU: 优化器状态 + 梯度 + 其余模型参数

Stage 3 + NVMe Offload:
  GPU: 模型参数[当前层]
  CPU: 部分缓冲
  NVMe: 优化器状态 + 梯度 + 模型参数（最慢但最省显存）
```

### 3.3 显存占用对比（以 7B 模型、4 张 GPU 为例）

| 方案 | 每张卡显存占用(估算) | 能否在 24GB 卡上跑 7B 全量？ |
|---|---|---|
| DDP | ~70-100 GB | ❌ 不可能 |
| ZeRO Stage 1 | ~50-70 GB | ❌ 仍不够 |
| ZeRO Stage 2 | ~30-50 GB | ❌ 勉强但危险 |
| ZeRO Stage 3 | ~10-20 GB | ✅ 可以（4 卡） |
| ZeRO Stage 3 + CPU Offload | ~5-10 GB | ✅ 轻松 |

### 3.4 优点

| 优点         | 说明                                 |
| ---------- | ---------------------------------- |
| **显存效率极高** | ZeRO-3 可训比 DDP 大 N 倍的模型（N = GPU 数） |
| **灵活**     | Stage 1/2/3 可切换，Offload 可配置        |
| **大模型必备**  | 70B+ 模型不用 DeepSpeed 基本训不了          |
| **微软维护**   | 质量高，与 PyTorch 生态集成好                |

### 3.5 缺点

| 缺点 | 说明 |
|---|---|
| **配置复杂** | 需要写 JSON 配置文件（`ds_config.json`） |
| **通信开销大** | ZeRO-3 的 all-gather 通信量远大于 DDP 的 all-reduce |
| **速度慢** | 同等条件下，每个 step 比 DDP 慢 10%~50% |
| **debug 难** | 报错信息不如 DDP 清晰 |

### 3.6 适用场景

- ✅ **模型太大，DDP 直接 OOM**（如 70B 在 8 张 40GB 卡上）
- ✅ **显存紧张**，想用更小的卡训更大的模型
- ✅ **不差那点速度**，优先考虑能不能跑起来
- ❌ **小模型多卡加速** → DDP 更快更简单，不需要 DeepSpeed

## 4. DDP vs DeepSpeed 核心对比表

| 维度 | DDP | DeepSpeed (ZeRO) |
|---|---|---|
| **开发方** | PyTorch 官方（Meta） | 微软 |
| **集成方式** | `torch.nn.parallel.DistributedDataParallel` | 需安装 `deepspeed` 包 + 配置文件 |
| **显存效率** | 低（全复制） | 高（分片，Stage 3 最彻底） |
| **通信模式** | all-reduce（梯度同步） | all-gather + reduce-scatter（参数+梯度） |
| **单步速度** | **快** | 慢（通信量大） |
| **最大可训模型** | 受限于单卡显存 × 卡数 | 受限于总显存（分片后大很多） |
| **配置难度** | 低（一行命令） | 中高（需写 ds_config.json） |
| **稳定性** | 高 | 中（版本兼容偶有坑） |
| **适合模型规模** | 能放进单卡（≤ 30B on A100） | 任意规模（特别适合 > 30B） |

## 5. 小模型（4B/8B）该怎么选？

这是你最关心的实操问题。直接给结论：

### 5.1 决策树

```
你的模型是 4B/8B，做 LoRA 微调
│
├─ 只有 1 张卡？
│  └─ 不需要分布式！直接选 none
│
├─ 有 2~4 张卡，想加速训练？
│  ├─ 每张卡 ≥ 24GB（3090/4090/A10）？→ 选 DDP ✅
│  └─ 每张卡 < 24GB（如 16GB 3080Ti）？→ 选 DeepSpeed ZeRO-2 或 ZeRO-3
│
└─ 有 8 张以上卡？
   └─ 选 DDP（简单快）或 DeepSpeed ZeRO-2（省显存），看具体卡型和模型大小
```

### 5.2 具体建议

| 场景 | 推荐方案 | 原因 |
|---|---|---|
| 4B LoRA，单卡 24GB | **none** | 单卡够用，不需要分布式 |
| 8B LoRA，单卡 24GB | **none** | 通常够，不够就降 batch size |
| 8B 全量微调，单卡 80GB(A100) | **none** | A100 80GB 够放 8B 全量 |
| 4B/8B LoRA，2~4 张 24GB 卡 | **DDP** | 简单快速，DDP 完美适用 |
| 8B 全量微调，2~4 张 24GB 卡 | **DeepSpeed ZeRO-3** | DDP OOM，ZeRO-3 分片后能跑 |
| 70B LoRA，任意卡配置 | **DeepSpeed ZeRO-3** | 模型太大，DDP 不可能 |

### 5.3 LLaMA Factory 里的配置对应关系

| LLaMA Factory 选项 | 底层对应 | 小模型建议 |
|---|---|---|
| `none` | 不走分布式 | 单卡训练 |
| `DDP` | `torch.nn.parallel.DistributedDataParallel` | **多卡小模型首选** |
| `DeepSpeed` | DeepSpeed ZeRO（默认 Stage 2，可在 `ds_config.json` 里改 Stage） | 模型大或显存不够时用 |
| `Horovod` | Horovod（老框架） | 一般不选 |

> [!tip] LLaMA Factory 里选了 DeepSpeed 后，ZeRO Stage 是几？
> LLaMA Factory 默认 DeepSpeed 配置通常是 **ZeRO Stage 2**。如果想用 Stage 3，需要自己改 `ds_config.json`（在 LLaMA Factory 的 `examples/deepspeed` 目录下有模板）。

## 6. 进阶：DeepSpeed ZeRO Stage 怎么选？

如果你决定用 DeepSpeed，该选 Stage 1 / 2 / 3？

| Stage | 什么时候选 | 通信开销 | 显存节省 |
|---|---|---|---|
| **Stage 1** | 优化器状态是瓶颈（大模型 + Adam） | 略高于 DDP | 中等（~4×） |
| **Stage 2** | 梯度也是瓶颈（中等模型） | 比 Stage 1 高 30% | 高（~8×） |
| **Stage 3** | 模型参数也是瓶颈（大模型，DDP OOM） | 最高（比 DDP 高 2~3×） | 最高（~N×） |
| **Stage 3 + CPU Offload** | GPU 显存仍不够 | 最高+CPU 带宽瓶颈 | 极大（模型放 CPU） |

**经验法则**：
- 能跑 Stage 2 就别用 Stage 3（速度差很多）
- 只有 Stage 2 也 OOM 时才上 Stage 3
- 只有 Stage 3 也 OOM 时才加 CPU Offload

## 7. 总结：一张图记住核心区别

```
DDP:  每张卡存 [完整模型 + 完整梯度 + 完整优化器]
       │
       │ 显存浪费大，但通信少、速度快
       ▼
     适合：模型能放进单卡，多卡只是加速

DeepSpeed ZeRO:
  Stage 1: 每张卡存 [完整模型 + 完整梯度 + 1/N 优化器]
  Stage 2: 每张卡存 [完整模型 + 1/N 梯度 + 1/N 优化器]
  Stage 3: 每张卡存 [1/N 模型 + 1/N 梯度 + 1/N 优化器]  ← 最省显存
       │
       │ 显存极省，但通信多、速度慢
       ▼
     适合：模型太大 DDP 跑不了，或显存非常紧张
```

**一句话记忆**：
- **DDP = 快但吃显存，小模型多卡加速用**
- **DeepSpeed = 慢但省显存，大模型/显存不够用**
