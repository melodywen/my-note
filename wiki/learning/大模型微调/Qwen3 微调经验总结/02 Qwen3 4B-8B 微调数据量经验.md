# 02 Qwen3 4B/8B 微调数据量经验

> 本篇聚焦一个实操高频问题：**用 Qwen3-4B 或 Qwen3-8B 这种中小模型做 LoRA 微调,到底需要多少条数据才合适？** 一句话结论：**3,000~5,000 条是性价比最高的甜区**,整体合理范围在几千到几万条之间。

## 1. 数据量的大致分档

| 数据量级 | 适用场景 | 效果预期 |
|---|---|---|
| **< 500 条** | 仅用于验证训练流程能跑通 | 模型几乎学不到东西,loss 可能还没收敛就结束了 |
| **500~2,000 条** | 极窄领域 + 模板化任务(如固定格式抽取) | 能学到基本模式,但泛化差、容易过拟合 |
| **3,000~5,000 条** | **大多数垂直场景的推荐起步量** | 模型能学到领域知识和回答风格,效果与成本的最佳平衡点 |
| **5,000~20,000 条** | 任务复杂度高、需要覆盖多种子场景 | 效果更稳定,泛化更好,但标注成本明显上升 |
| **20,000~50,000 条** | 多任务混合训练、通用能力保持 | 接近中等规模 SFT,需要更多算力和更细的数据配比策略 |
| **> 50,000 条** | 4B/8B 小模型很少需要这么多 | 容易过拟合、训练时间长,投入产出比下降;此时应考虑换更大模型 |

## 2. 为什么 3,000~5,000 条是甜区？

### 模型容量角度

4B/8B 参数量的模型"记忆容量"有限——LoRA 只训练很小一部分参数(rank=8 时约占全参数的 0.1%~0.3%),能有效学到的新知识量本身就有上限。3,000~5,000 条数据刚好能:
- 覆盖目标领域的核心模式和常见变体
- 不至于多到让模型"记住答案"(过拟合)而丧失泛化

### 训练稳定性角度

- 数据太少(< 1,000):loss 波动大,模型容易过拟合到训练集的特定表述
- 3,000~5,000 条:配合 1~3 个 epoch,loss 能平稳下降到合理水平
- 数据太多(> 30,000):小模型可能在后期 loss 不再下降,白费算力

### 实操成本角度

- 高质量数据标注是最贵的环节(比算力贵)
- 3,000 条人工精标 ≈ 1~2 周工作量(单人)
- 投入产出比远优于堆到几万条

## 3. 影响数据量选择的关键因素

不是所有场景都适用"3,000~5,000"这个数字,要根据以下因素调整:

### 任务复杂度

| 任务类型 | 建议数据量 | 原因 |
|---|---|---|
| 单一格式抽取(如从文本提取日期) | 1,000~3,000 | 模式单一,少量数据就够 |
| 客服对话 / 问答 | 3,000~5,000 | 需覆盖常见问题变体 |
| 多轮推理 / 复杂指令跟随 | 5,000~15,000 | 子场景多、推理链路长 |
| 多任务混合(做分类+做生成+做抽取) | 10,000~30,000 | 每个子任务都需要足够样本 |

### 领域与基座模型的"距离"

- **领域离基座近**(如通用问答、翻译):基座已有相关能力,少量数据(2,000~3,000)就能"唤醒"
- **领域离基座远**(如医疗报告、法律条文、行业黑话):需要更多数据(5,000~10,000)教会模型新概念

### 数据质量 vs 数据数量

> [!important] 质量永远比数量重要
> **1,000 条高质量、精标注的数据,效果通常优于 10,000 条低质量、自动生成的数据。**
>
> "质量"包括:
> - 答案准确、无事实错误
> - 格式统一、符合期望输出
> - 覆盖了边界情况和难例
> - 没有大量重复/近似样本

## 4. 数据量与训练参数的配合

数据量定了之后,训练参数也要跟着调:

| 数据量 | 建议 epoch | 建议 effective batch size | 原因 |
|---|---|---|---|
| 1,000~3,000 | 2~3 | 8~16 | 数据少,多看几遍加深学习 |
| 3,000~5,000 | 1~2 | 16~32 | 甜区,1~2 遍通常够 |
| 5,000~20,000 | 1 | 16~32 | 数据够多,1 遍防过拟合 |
| > 20,000 | 1(甚至 < 1,即不跑完) | 32~64 | 数据充足,跑完一遍都可能过拟合 |

> [!tip] 怎么判断数据量是否够了？
> - **训练 loss 在 1 个 epoch 结束时已平稳** → 数据量够了,不需要更多
> - **训练 loss 还在快速下降就结束了** → 可以加 epoch 或加数据
> - **验证 loss 先降后升(过拟合)** → 数据量够了,减少 epoch 或加正则

## 5. 实操建议：渐进式加数据

不要一开始就堆大量数据。推荐的流程:

1. **先用 500 条跑通流程** — 确认训练链路无误(参考 [[03 ms-SWIFT 测试运行示例]])
2. **用 1,000~2,000 条做第一版** — 快速看到模型行为变化,评估方向对不对
3. **扩到 3,000~5,000 条做正式版** — 精选数据,覆盖核心场景,这通常就是交付版本
4. **只有明确需要时才继续加** — 如果 3K~5K 效果已达标,就不要盲目堆量

## 6. 小结

| 要点 | 内容 |
|---|---|
| 4B/8B 微调的甜区数据量 | **3,000~5,000 条** |
| 合理范围 | 几千到几万条(视任务复杂度) |
| 核心原则 | 质量 > 数量;渐进式增加;配合 loss 曲线判断 |
| 超过 5 万条的场景 | 小模型很少需要,考虑换更大模型 |
