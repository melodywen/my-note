# 06 RLHF 人类对齐详解

> WebUI 顶部那个 **「LLM人类对齐」** 标签(对应命令 `swift rlhf`),很多人看到 DPO/KTO/PPO 一堆缩写就懵了。本篇只讲清楚三件事:**它是什么、为什么需要它、ms-SWIFT 里这几种方法怎么选**。

## 1. 一句话:让模型"说人话、合人意"

普通微调(SFT,就是你前面做的 `swift sft`)教模型的是:**"这个问题的标准答案长这样"**。

但有些东西没有唯一标准答案,而是**偏好问题**:

- 同一个问题,A 回答和 B 回答都"对",但人类觉得 A 更礼貌、更安全、更有帮助。
- "帮我写个删库脚本"——模型该照做,还是该委婉拒绝?
- 两段都通顺的回答,哪段语气更好?

> [!key-insight] SFT 教"对不对",RLHF 教"好不好"
> - **SFT(监督微调)**:给模型看大量"问题→标准答案"对,让它学会**模仿**。
> - **RLHF(基于人类反馈的强化学习)**:给模型看"对于同一问题,人类更喜欢哪个回答",让它学会**对齐人类偏好**。
>
> RLHF = **R**einforcement **L**earning from **H**uman **F**eedback,基于人类反馈的强化学习。"人类对齐"就是它的中文叫法。

## 2. 为什么需要它?——SFT 的天花板

光靠 SFT 有几个治不了的毛病:

| SFT 的局限                  | RLHF 怎么补             |
| ------------------------ | -------------------- |
| 只会模仿给的答案,学不到"什么样的回答更受欢迎" | 直接用人类偏好(A 比 B 好)来调   |
| 容易学到"正确但生硬/啰嗦/不安全"的表达    | 把"有用、无害、诚实"这类软性目标灌进去 |
| 没法表达"宁可不答也别答错"           | 偏好数据里可以体现拒绝、谨慎       |

> [!important] <font color="red">现实中的流程通常是:先 SFT 打底,再 RLHF 精修</font>
> 业界主流大模型(ChatGPT、Qwen-Chat 等)几乎都是 **SFT → RLHF** 两段式训出来的。SFT 让模型"会答",RLHF 让模型"答得讨人喜欢、守规矩"。
>
> **对你现在(入门、Mac、小模型验证)来说:RLHF 是进阶内容,先把 SFT 跑通理解透,RLHF 知道是干嘛的即可,不急着上手。**

## 3. 偏好数据长什么样

RLHF 的"燃料"不再是"问题+答案",而是**"问题 + 一个更好的答案 + 一个更差的答案"**:

```json
{
  "messages": [{"role": "user", "content": "介绍一下你自己"}],
  "chosen":   "你好!我是一个 AI 助手,很高兴为你服务,有什么可以帮你?",
  "rejected": "我是AI。"
}
```

- `chosen`:人类**更喜欢**的回答(更好)
- `rejected`:人类**不喜欢**的回答(更差)

模型从大量这种对比里学会:**朝 chosen 那种风格靠拢,远离 rejected 那种**。

## 4. ms-SWIFT 支持的几种方法(`--rlhf_type`)

`swift rlhf` 通过 `--rlhf_type` 选具体算法。常见这几种,按"好不好上手"排序:

| 方法       | 全称 / 含义                               | 数据要求                 | 特点                                                       | 上手难度  |
| -------- | ------------------------------------- | -------------------- | -------------------------------------------------------- | ----- |
| **DPO**  | Direct Preference Optimization 直接偏好优化 | chosen + rejected 成对 | <font color="red">最主流、最常用</font>。不用单独训奖励模型,直接拿偏好对优化,简单稳定 | ⭐ 容易  |
| **KTO**  | Kahneman-Tversky Optimization         | **单条**好/坏标注即可(不用成对)  | 数据更好收集(只标"这条好"或"这条差"),适合没有成对数据时                          | ⭐ 容易  |
| **ORPO** | Odds Ratio Preference Optimization    | chosen + rejected    | 把 SFT 和偏好优化**合二为一**,一步到位                                 | ⭐⭐ 中等 |
| **PPO**  | Proximal Policy Optimization 近端策略优化   | 需先训**奖励模型**          | 经典 RLHF(ChatGPT 早期用的),效果好但**流程复杂**:要训奖励模型+多模型同时跑         | ⭐⭐⭐ 难 |
| **GRPO** | Group Relative Policy Optimization    | 带可验证奖励(如数学对错)        | 强化学习,擅长**推理/数学**类任务,WebUI 里单独一个标签                        | ⭐⭐⭐ 难 |

> [!key-insight] 新手只需记住 DPO
> - **想做对齐、又想省事** → 用 **DPO**,这是目前性价比最高、文档最多的选择。
> - **手头只有"单条好/坏"标注、凑不出成对** → 用 **KTO**。
> - **PPO/GRPO** 是重武器,涉及奖励模型和强化学习,入门阶段了解概念即可,别一上来就碰。

## 5. 命令长什么样(以 DPO 为例)

和 `swift sft` 几乎一样,只是命令变 `rlhf`、多了 `--rlhf_type`:

```bash
swift rlhf \
  --rlhf_type dpo \
  --model Qwen/Qwen3-0.6B \
  --tuner_type lora \
  --dataset <你的偏好数据集> \
  --torch_dtype float32 \
  --output_dir output
```

> [!warning] Mac 上同样的注意事项
> 和 SFT 一样:Mac 走 MPS/CPU、用 `float32`、`--tuner_type lora`(不是 `--train_type`)、`--dataloader_num_workers 0`。这些坑见 [[03 ms-SWIFT 测试运行示例]]。
>
> RLHF 比 SFT 更吃显存(DPO 要同时跑当前模型 + 参考模型两份),小机器上更要用小模型 + LoRA。

## 6. 为什么 GRPO 要单独拎一个标签?

WebUI 顶部除了「LLM人类对齐」,还单独有个 **「LLM GRPO」** 标签。明明 GRPO 本质也是 `swift rlhf --rlhf_type grpo`(RLHF 家族一员),为什么不放进"人类对齐"页,而要单开?

### 直接原因:GRPO 有一堆别人没有的独占配置

打开 GRPO 页会看到 DPO/KTO 页**根本没有**的区块:

| GRPO 页特有的东西                      | DPO/KTO 有吗 | 干嘛的                          |
| -------------------------------- | ---------- | ---------------------------- |
| **奖励函数 `--reward_funcs`**        | ❌ 没有       | GRPO 核心:用函数/规则给回答打分(如数学答对给分) |
| **奖励函数权重 `--reward_weights`**    | ❌ 没有       | 多个奖励函数怎么配比重                  |
| **「奖励模型设置(GRPO高级参数)」整块**         | ❌ 没有       | 一整组只属于 GRPO 的高级项             |
| System 里那段 `<think>/<answer>` 模板 | 一般不用       | GRPO 常用于训推理,引导模型先想再答         |

> [!key-insight] 单独成页的根本原因:它要填"奖励函数"
> DPO/KTO 喂的是"哪个回答更好"的**现成偏好数据**,填个数据集路径就完事。
> 而 **GRPO 要你定义"怎么自动给回答打分"**(奖励函数 + 权重),这是它独有的一大块配置。两者表单字段差异太大,塞一起会很乱,所以 UI 把 GRPO 拆出来单独一页。

### 深层原因:学习机制根本不同

| | DPO / KTO(在"人类对齐"页) | GRPO(单独页) |
|---|---|---|
| 学习信号 | 现成的**人类偏好对**(chosen/rejected) | **奖励函数实时打分**(模型自己生成多个答案,按分高低学) |
| 像什么 | "老师给了标准对比,照着学" | "你做一堆题,系统自动判分,朝高分进化" |
| 擅长 | 调语气、安全、有用性 | <font color="red">数学、代码、逻辑推理</font>这类**有明确对错**的任务 |
| 配置复杂度 | 给个偏好数据集就行 | 要定义奖励函数、权重、采样组等一大堆 |

### 为什么最近这么受重视?

> [!important] GRPO 是 DeepSeek-R1 那套"推理模型"训练的核心方法
> 2025 年 DeepSeek-R1 火了之后,GRPO 成了训练"会思考、会推理"模型的当红方法,能让小模型在数学/代码上大幅提升。ms-SWIFT 顺应热点,把它单独做成一个标签,方便专门调它那一堆参数。

> [!warning] GRPO 是进阶中的进阶,入门别急着碰
> 它涉及奖励函数设计、强化学习采样,配置和调参都复杂,还很吃算力。**入门阶段知道"它是用自动打分来训推理能力的"即可**,先把 SFT、再把 DPO 理解透。
