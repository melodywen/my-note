# 大模型微调概述

## 一句话定义

微调(Fine-tuning):拿一个已经预训练好的大模型,用你自己的标注数据**继续训练**,改动它的参数,让它在特定任务/领域上表现更好。

## 为什么需要微调

预训练模型(如 GPT、Qwen 的基座)是在海量通用语料上训出来的,它"什么都懂一点",但有两类问题靠提示词解决不了:

1. **目标不一致**:预训练学的是"预测下一个词",而你要的是"做情感分类""按固定格式输出 JSON"。能力方向对不上。
2. **领域分布不同**:模型见过的多是通用网页文本,没见过你公司的医疗病历、法律合同、内部术语。它对你的领域"水土不服"。

> [!key-insight] 核心区别:微调 vs RAG/Agent
> - **RAG / Agent**:模型本体不动,在**外面**加检索库、加工具链 —— 像给一个人配了本字典和一套工具。
> - **微调**:直接改**模型参数** —— 像让这个人把知识/技能"背进脑子里",永久掌握,不依赖外部。

## 微调到底改了什么(原理)

这是关键,也是之前那版没讲的部分。

模型的"能力"本质上存在它的**参数(权重)**里。微调做的事:

```
预训练权重 W(通用能力)
   ↓  用你的标注数据做反向传播,更新权重
微调后权重 W'(通用能力 + 你的任务能力)
```

具体机制——**知识迁移**:

- **底层参数**:保留通用特征(词义、语法、句子结构),基本不大动。
- **顶层参数**:通过反向传播重点调整,去适配你的任务。

打个比方:一个学了十年通识教育的大学生(预训练),你让他去做三个月的法律实习(微调)。他不用重新学认字、逻辑(底层能力还在),只是把通用能力"特化"到法律场景(顶层调整)。

但这里藏着一个代价:如果实习太猛、只练法律,他可能把别的领域知识忘了 —— 这就是 [[灾难性遗忘]],微调的头号风险。

## 一个具体例子:把通用模型变成客服助手

假设你要做一个电商客服模型,流程是这样的:

**1. 准备数据**(微调的成败关键,见 [[SFT 与指令微调]]):
```json
{"instruction": "用户问:我的订单什么时候发货?", "output": "您好~ 订单通常在付款后 24 小时内发货,可在『我的订单』查看物流。还有什么能帮您?"}
{"instruction": "用户问:可以退货吗?", "output": "当然可以哦~ 7 天无理由退货,商品保持完好即可。需要我帮您发起吗?"}
```
注意:这里教的不只是"知识",更是**说话的风格**(亲切、带语气词、主动追问)。

**2. 选微调方式**(见 [[全量微调与高效微调]]):
- 数据几千条、显卡只有一张 → 用 [[LoRA 与 QLoRA|LoRA]],只训练少量新增参数,省钱省显存。
- 要做深度改造、资源充足 → 全量微调。

**3. 训练 + 评估**:用低学习率训练,训完后既要测"客服任务做得好不好",也要测"通用能力有没有退化"。

**结果**:模型"原生"就会用客服口吻回答,不需要每次都在提示词里写一长串"请用亲切的语气……"。

## 什么时候用微调,什么时候用 RAG?

| 你的需求                              | 推荐方案        |
| --------------------------------- | ----------- |
| 改变模型**说话风格/行为**(如客服口吻、固定输出格式)     | 微调          |
| 让模型掌握一种**通用任务能力**(如分类、特定推理)       | 微调          |
| 让模型回答**实时/海量/经常变动的知识**(如最新文档、产品库) | RAG         |
| 知识量巨大、且更新频繁,不想反复训练                | RAG         |
| 既要专业行为、又要最新知识                     | 微调 + RAG 组合 |

> [!key-insight] 判断口诀
> 要"新行为/新风格" → 微调;要"新知识(尤其常变)" → RAG。

## 常见误区

- **数据太少**:几十条就想微调,极易过拟合,模型只会背这几条,不会举一反三。
- **学习率太高**:参数被改得太猛,通用能力崩塌(灾难性遗忘)。一般用 1e-5 ~ 5e-5。
- **指望微调"灌知识"**:微调擅长改行为/风格,对"记住大量事实性知识"效果不稳定,这类需求 RAG 往往更合适。
